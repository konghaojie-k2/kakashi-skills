# 卡卡西 Code

AI驱动的代码风格学习与顾问。通过Subagent辩论帮助用户做出最佳风格决策。

## 触发条件

当用户请求以下任一操作时使用此技能：
- 学习/分析当前项目代码风格
- 检测代码风格冲突
- 统一代码风格
- 选择融合还是覆盖现有风格

## 核心能力

1. **风格学习** - 扫描项目代码，提取风格特征
2. **冲突检测** - 发现新代码与现有风格冲突
3. **辩论机制** - 启动Subagent辩论，融合派 vs 覆盖派
4. **用户裁决** - 让用户选择最终方案

---

## 工作流程

### 阶段1：风格学习

请按以下步骤分析项目代码风格：

1. **扫描项目**
   - 读取项目根目录及 src/ 下的代码文件
   - 优先选择 5-10 个代表性文件

2. **提取风格特征**
   对每个文件分析：
   - 命名风格：`snake_case` / `camelCase` / `PascalCase`
   - 类型注解：有 / 无 / 部分
   - 文档风格：`Google` / `NumPy` / `Sphinx` / 无
   - import 排序：字母 / 长度 / 分组
   - 缩进：空格数 (2/4)
   - 引号：单引号 / 双引号
   - 分号：使用 / 不使用
   - 其他特色规则

3. **输出风格报告**

```
【风格分析报告】
| 特征 | 当前风格 |
|------|---------|
| 命名 | snake_case |
| 类型注解 | 有 (部分) |
| 文档 | Google Style |
| ... | ... |

请确认以上分析是否正确，或补充遗漏的风格特征。
```

### 阶段2：冲突检测

当有新代码需要与现有风格对齐时：

1. **对比分析**
   - 列出新代码与现有风格的差异点
   - 每个差异标注：轻微 / 中等 / 严重

2. **生成选项**

```
【风格冲突检测】
差异点：
1. 命名风格：新代码用 camelCase，项目用 snake_case (严重)
2. 文档：新代码无docstring (中等)

建议方案：
- 方案A：融合 - 保持现有风格，将新代码改为 snake_case
- 方案B：覆盖 - 采用新代码风格，更新项目配置
- 方案C：部分融合 - 命名用新风格，其他保持现有

请选择方案，或输入自定义方案。
```

### 阶段3：Subagent辩论

当用户需要深入分析时，启动辩论模式：

```
【启动风格辩论】

我将同时启动两个Subagent：
- 👤 融合派：主张保持现有风格
- 👤 覆盖派：主张采用新风格

请稍候...
```

**融合派提示词**：

```
你代表"融合派"，你的任务是：
1. 分析当前项目风格特征
2. 列出保持现有风格的理由（一致性、可维护性、团队共识）
3. 给出将新代码融合到现有风格的具体方案
4. 反驳覆盖派的观点

请用简洁有力的论述（不超过100字）支持融合方案。
```

**覆盖派提示词**：

```
你代表"覆盖派"，你的任务是：
1. 分析新代码的风格优势
2. 列出采用新风格的理由（现代化、更佳实践、减少技术债）
3. 给出覆盖现有风格的具体方案
4. 反驳融合派的观点

请用简洁有力的论述（不超过100字）支持覆盖方案。
```

### 阶段4：用户裁决

辩论结束后，总结并请求裁决：

```
【辩论总结】

🔵 融合派观点：
- 保持项目一致性
- 减少团队学习成本
- 具体方案：新代码改用 snake_case + 添加 Google Style docstring

🟢 覆盖派观点：
- 新代码风格更现代化
- 减少未来迁移成本
- 具体方案：更新 .kakashi.json 配置，新代码统一用 camelCase

请选择：
A. 融合派方案
B. 覆盖派方案
C. 自定义方案（请说明）
```

## 风格配置保存

确认风格后，生成 `.kakashi.json`：

```json
{
  "version": "1.0",
  "language": "python",
  "naming": "snake_case",
  "types": "annotations",
  "docstring": "google",
  "imports": "isort",
  "indent": 4,
  "quotes": "double"
}
```

---

## 冲突处理原则

| 冲突类型 | 建议 |
|----------|------|
| 命名风格不一致 | 优先融合（改新代码） |
| 文档风格不一致 | 可部分融合 |
| 新代码显著优于现有 | 建议覆盖 |
| 团队有明确规范 | 遵守团队规范 |

---

## 注意事项

1. **尊重现有风格** - 不要强行推广新风格
2. **给出明确选项** - 避免让用户做开放式选择
3. **提供具体方案** - 每种选择都要有可执行方案
4. **说明理由** - 让用户理解决策背后的逻辑
5. **用户优先** - 最终决策权交给用户
